# ТЗ: админка Razbor 2.0 — командный центр

Сейчас /admin показывает данные, но не помогает принимать решения: техно-термины,
сломанные проценты (>100%), мёртвая таблица лидов, нельзя посмотреть аудит лида.
Переделать в рабочий инструмент. Делать по частям, показывать скрины, ждать «ок».

> **Обновления в этой версии:**
> - A3 переписана: метка владельца через COOKIE вместо IP (у хозяйки VPN, IP
>   меняется). Авто-метка при входе в /admin + кнопка-страховка + фильтр «без
>   моих тестов» ВКЛ по умолчанию.
> - Добавлена Часть G: фичи из топовых сервисов (Mixpanel/Cometly/Funnel.io),
>   отобранные под наш масштаб — источник трафика (UTM), время в воронке,
>   путь одного лида. Без enterprise-машинерии.

## Часть A. ФИКС БАГОВ ВОРОНКИ (сначала)

**A1. Проценты >100% — главный баг.** Сейчас «Увидели тизер 220%», «Открыли разбор
   225%». Воронка считает СОБЫТИЯ, а не уникальных людей (один человек открыл
   разбор 5 раз = 5). Починить: на каждом шаге считать УНИКАЛЬНЫЕ (audit_id ?? meta.ip
   или distinct session). Проценты только убывающие, никогда не >100%. Каждый
   шаг считается от предыдущего.

**A2. Фильтр по датам** сверху: Сегодня / 7 дней / 30 дней / Всё время.
   По умолчанию — 7 дней. Вся воронка и лиды считаются за выбранный период.
   (Бонус: старый тестовый шум сам выпадает из окна за пару дней.)

**A3. Метка владельца через COOKIE (не IP — у хозяйки VPN, адрес меняется).**
   IP-список ненадёжен, поэтому основной механизм — cookie:
   1. Авто-метка: при успешном входе в /admin под Basic-auth middleware ставит
      браузеру cookie is_owner=1 (path=/, на год, httpOnly, SameSite=Lax).
      Сам факт «прошёл Basic-auth = владелец» помечает браузер. Нажимать ничего
      не надо, от IP не зависит.
   2. Клеймо на событиях: /api/events читает cookie (уходит автоматически на тот
      же домен) → проставляет meta.is_owner=true. Все заходы с этого браузера
      помечаются на лету.
   3. Фильтр: по умолчанию события с meta.is_owner исключаются (галочка «без моих
      тестов» = ВКЛ). Переключатель ?owner=1 показывает всё, включая мои.
   4. Кнопка-страховка «не считать этот браузер» / «считать» — если зайду с нового
      устройства и захочу пометить вручную (ставит/снимает тот же cookie).
   - Честное ограничение: cookie метит только ВПЕРЁД. Прошлые ~61 анонимный заход
     уже без метки, ретроактивно не вычислить (IP нестабилен, session_id в схеме
     нет). Закрывается фильтром дат A2 — старый шум уходит из окна сам.
   - OWNER_IPS оставить как запасной механизм в .env (не основной).

## Часть B. ЧЕЛОВЕЧЕСКИЙ ЯЗЫК (убрать техно-термины)

Везде в админке заменить жаргон на понятные слова владельцу:

| Было (термин)        | Стало (по-человечески)                         |
|----------------------|------------------------------------------------|
| Увидели тизер        | Посмотрели короткий разбор                      |
| Открыли разбор       | Открыли полный разбор                           |
| engaged              | Откликнулся (написал/кликнул)                   |
| new                  | Новый                                           |
| leadgen              | Сбор заявок                                      |
| local                | Местный бизнес                                   |
| канал: phone         | Телефон                                          |
| канал: telegram      | Telegram                                         |
| sms:gift (skipped)   | СМС-подарок (не отправлен)                       |
| telegram:gift        | Подарок в Telegram отправлен                     |
| events               | События / Действия                               |
| drop-off             | Где отваливаются                                 |

Тип сайта (local/leadgen/ecom и пр.) — переводить на русский понятно.
Статусы доставки — словами, что реально произошло.
Нигде не оставлять английских технических ключей в видимом тексте.

## Часть C. ГЛАВНЫЕ ЦИФРЫ СВЕРХУ (карточки)

Над воронкой — ряд крупных карточек (brutalism, по дизайн-системе оксблад/бумага),
самое важное одним взглядом, за выбранный период:
- Всего заявок (лидов)
- Новых за период
- Откликнулись (написали/кликнули)
- Конверсия «зашёл → оставил заявку» (%)
- Средний балл проверенных сайтов (низкий = больше горячих для редизайна)

## Часть D. ТАБЛИЦА ЛИДОВ — СДЕЛАТЬ РАБОЧЕЙ

Сейчас мёртвый список. Добавить действия:
1. **Открыть аудит лида** — ссылка/кнопка на /a/[id] (или встроенный просмотр),
   чтобы видеть КАКОЙ сайт человек проверял и ЧТО ему показали. Это ключевое —
   хочу смотреть разбор каждого, кто оставил заявку.
2. **Сменить статус** вручную: Новый → Написал → Ответил → Клиент → Отказ.
   (dropdown, пишется в leads.status; не ломать авто-engaged по вебхуку).
3. **Сортировка** по баллу и по дате. По умолчанию — новые сверху.
4. **Подсветка горячих**: лиды с баллом аудита < 50 — пометить (оксблад-точка
   или тег «горячий»), это самые перспективные для продажи редизайна.
5. Колонки по-человечески: Контакт / Канал / Статус / Тип сайта / Балл / Что
   отправлено / Когда. Балл — крупно, цветом (низкий красным = есть что чинить).

## Часть E. AI-АНАЛИЗ МАРКЕТОЛОГА (кнопка)

Кнопка «🔍 Разобрать статистику» (запуск по клику, НЕ автоматически — экономим
токены). По нажатию: собрать текущие цифры (воронка, конверсии, баллы, лиды за
период) → отправить в OpenAI (gpt-5.4-mini, дёшево) с промптом «ты маркетолог-
аналитик, посмотри на воронку сервиса и скажи человеческим языком: что хорошо,
где главные дыры, где люди отваливаются и почему, что чинить в первую очередь» →
показать ответ в блоке под кнопкой.

Требования к AI-анализу:
- Тон по docs/VOICE.md (живой эксперт, без жаргона, прямо, без воды, без тире-
  частокола). Это важно — анализ должен звучать как живой маркетолог, не как бот.
- Конкретика по ЦИФРАМ: «из 61 зашедшего заявку оставили 4 — это 7%, узкое место
  на шаге ввода URL, теряете там почти всех». Не общие советы.
- Структура: 1) что работает, 2) где главная дыра, 3) что чинить первым.
- Честность: если данных мало (как сейчас, в основном тесты) — так и сказать,
  «выводов мало, нужен реальный трафик», не выдумывать.
- Кэшировать результат (не гонять OpenAI повторно, пока не нажали снова).
- Стоимость одного анализа показать мелко (≈X¢), чтобы понимать траты.

## Часть F. РЕАЛ-ТАЙМ (опционально, обсудить)

Хозяйка хочет «отслеживать в реальном времени». Варианты (выбрать на этапе плана):
- Простой автообновление страницы раз в 30-60 сек (дёшево, без сокетов).
- Бейдж «N новых заявок с последнего захода».
- Не делать live-сокеты (избыточно для текущего трафика).
Рекомендую: автообновление цифр раз в 60с + бейдж новых заявок. AI-анализ —
только по кнопке, не в автообновлении.

## Часть G. ИЗ ТОПОВЫХ СЕРВИСОВ (что реально полезно под наш масштаб)

Отобрано из Mixpanel/Cometly/Funnel.io — только то, что подходит соло-сервису,
без избыточной enterprise-машинерии (мульти-тач атрибуция, MMM, data warehouse —
НЕ делаем, это для больших команд).

1. **Источник трафика (откуда пришёл лид)** — сейчас этого нет вообще, а это
   критично. Ловить UTM-метки из ссылок (?utm_source=telegram / instagram /
   email / direct), писать в events/lead. В админке: разбивка воронки и лидов
   по источнику. Поймёшь какой канал реально приводит заявки, а не сжигает время.
   Для своих ссылок (бот, посты) проставлять utm_source. По-человечески:
   колонка/фильтр «Откуда пришёл».

2. **Время в воронке** — сколько проходит от «зашёл» до «оставил заявку», и от
   заявки до отклика на письма. Показать средние («обычно оставляют заявку за
   N минут», «откликаются за N дней»). Видно где люди зависают.

3. **Путь одного лида (журнал)** — при открытии лида показать его историю
   событиями по-человечески: «18:30 зашёл с Telegram, 18:31 проверил site.ru,
   18:33 открыл полный разбор, 18:34 оставил контакт, день 2: открыл письмо,
   кликнул». Усиливает просмотр аудита: видно не только ЧТО проверял, но и весь
   путь. Без сокетов, просто из таблицы events по lead/session.

Эти три — заметное усиление без перегруза. Всё на наших данных (events),
новых интеграций не нужно.

## Приёмка
- Проценты воронки не >100%, считаются уники, мои тесты исключаются.
- Фильтр дат работает. Карточки главных цифр сверху.
- Нет техно-терминов в видимом тексте (всё по таблице из части B).
- Из таблицы лида можно открыть его аудит и сменить статус.
- Горячие лиды (балл <50) подсвечены.
- Кнопка AI-анализа даёт живой разбор по VOICE.md, кэшируется, показывает цену.
- Дизайн по REDESIGN.md (brutalism, оксблад/бумага, без слопа).
- tsc+build зелёные, /admin под Basic-auth как раньше.

## Что НЕ трогать
- Логику аудита, гейт, лимиты, воронку follow-up.
- Basic-auth механизм (он теперь работает).
- Схему БД менять минимально (статусы лида уже есть; для is_owner/session —
  по возможности без тяжёлой миграции, можно по IP-списку в env).
