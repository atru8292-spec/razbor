# ТЗ v7: Razbor — сервис AI-аудита сайтов (финал, self-hosted)

Документ для Claude Code. Заменяет v1–v6. Всё приложение живёт на одном выделенном VPS, без Vercel и без serverless-ограничений. База данных на managed Supabase. Скрапер свой (файл scraper.js).

Что добавлено в v5 (по итогам анализа живых конкурентов): оценка потерь в деньгах/процентах, опциональный вопрос про цель сайта, адаптация лензы под тип сайта, направление «ИИ-видимость (AEO)», аннотированные скриншоты, выгрузка отчёта в PDF. Контакт берётся один раз — PDF и подарок отдаются как часть награды за этот один контакт, повторно номер не спрашиваем.

Что добавлено в v6: транспорт почты Resend → Unisender Go (только отправка по API/SMTP, дёшево, РФ-доставка, ФЗ-152, серверы в РФ). Шаблоны писем, цепочка follow-up (крон в воркере), таблица событий `events` и закрытая страница `/admin` с воронкой и конверсией — всё in-house, своими руками. Арендуем только трубу для доставки, остальное наше.

Что изменилось в v7: весь ИИ — на OpenAI напрямую (ключ владельца, оплачиваемый из РФ, вызовы идут с немецкого VPS). Vision-аудит на GPT (рабочая модель класса GPT-5.4, при нехватке качества GPT-5.5), мелочь (тип сайта, ниша, AEO) на mini/nano. Поиск конкурентов — встроенный веб-поиск OpenAI (web search tool в Responses API, с источниками), Perplexity больше не обязателен, остаётся опциональным апгрейдом. OpenRouter не используем (в РФ заблочен и не нужен). Claude (через kie.ai) держим в уме как запасной «мозг», если GPT на тесте даёт слишком общую выдачу — по цене Claude и GPT равны, так что это решение по качеству, не по деньгам.

## 0. Суть и цель

Название сервиса: **Razbor** (пишется латиницей, читается как «разбор»). Логотип и заголовки капсом: RAZBOR. Слоган-ориентир: «Видно, где сайт теряет заявки».

Сервис по ссылке делает экспертный аудит сайта и отвечает на вопрос: где и почему сайт теряет заявки. Плюс находит конкурентов и показывает, что есть у них, но нет у этого сайта. Это лид-магнит и витрина: человек получает пользу и за контакт получает подарок, а «как чинить» это платная услуга владельца (редизайн и автоматизация).

Два требования к качеству: выдача экспертная и конкретная (привязана к элементам сайта, мыслит деньгами), и сам сервис выглядит на уровне (не нейро-шаблон).

## 1. Инфраструктура (где что крутится)

Один выделенный VPS (Xorek Cloud, отдельный от n8n), Ubuntu 22.04/24.04, минимум 2 vCPU и 4 ГБ RAM (комфортно 8), плюс 2–4 ГБ swap. На нём через pm2 крутятся:
- Next.js-приложение (фронт + API), запуск `next start` за nginx.
- Фоновый воркер аудитов (отдельный процесс pm2).
- Скрапер на Playwright (файл scraper.js, локально на этом же сервере).
- Телеграм-бот (для выдачи подарка).

nginx как реверс-прокси, HTTPS через Let's Encrypt (как ты уже делаешь для neiro-arina.ru), отдельный домен или поддомен под Razbor (проверить свободный: getrazbor, razbor.pro, razbor.site, tryrazbor). Деплой: git pull, `npm run build`, `pm2 restart`.

Внешние сервисы: managed Supabase (БД), OpenAI (vision-аудит + встроенный веб-поиск для конкурентов; мелочь на mini/nano), Unisender Go (транспорт почты, только отправка по API/SMTP — РФ-серверы, ФЗ-152), СМС-провайдер для РФ (опционально), Cloudflare Turnstile (анти-бот). Perplexity — опционально (можно заменить им веб-поиск OpenAI ради качества). Конструктор писем, сегментацию и аналитику у Unisender НЕ берём — это пишем сами.

Vercel и QStash не используются: на постоянном сервере фоновый процесс снимает все ограничения по времени.

## 2. Пользовательский флоу

1. Лендинг: поле URL, понятное обещание, кнопка «Проверить сайт».
2. Перед запуском: телефон или телеграм (почта по желанию), галочка согласия, Turnstile. Обещаем подарок. Плюс одно НЕобязательное поле «главная цель сайта» (заявки / продажи / запись / подписка — выпадающий список или короткий ввод). Если заполнено, аудит точнее судит релевантность и message-match.
3. Проверка лимита: один бесплатный аудит на человека в месяц (раздел 10). Если уже был, мягко сообщаем.
4. Запуск: создаётся задача, фронт показывает экран прогресса с этапами и живым превью скриншота их сайта. Опрашивает статус раз в 2–3 сек (или SSE).
5. Результат: тизер сразу (две оценки и топ-3 точки потери), полный разбор открыт, контакт оставлен.
6. Подарок и отчёт: ссылка на подарок видна на экране сразу, плюс уходит по каналу контакта (раздел 11).
7. В конце один CTA на эксперта.
8. Результат по уникальной ссылке, делится с красивой OG-карточкой (раздел 13).
9. Через несколько дней follow-up (раздел 12).

## 3. База данных (Supabase) + SQL

```sql
create table audits(
  id uuid primary key default gen_random_uuid(),
  url text not null,
  goal text,                                  -- опциональная цель сайта со стартовой формы
  site_type text,                             -- авто: ecommerce|leadgen|saas|info|local
  status text not null default 'pending',   -- pending | running | done | error
  progress text,
  result jsonb,
  screenshots jsonb,                          -- ссылки или ключи на скриншоты
  pdf_url text,                               -- ссылка на сгенерированный PDF отчёта
  error_message text,
  created_at timestamptz default now()
);
create table leads(
  id uuid primary key default gen_random_uuid(),
  audit_id uuid references audits(id),
  phone text, telegram text, email text,
  channel text,                               -- основной канал контакта
  consent boolean default false, consent_at timestamptz,
  status text default 'new',                  -- new | engaged | replied | client (этап воронки)
  followup_stage int default 0,               -- какое касание цепочки уже отправлено (0..3)
  ip text, source text,
  created_at timestamptz default now()
);
create table emails_log(
  id uuid primary key default gen_random_uuid(),
  lead_id uuid references leads(id),
  channel text,                               -- telegram | sms | email
  type text,                                  -- gift | report | followup
  status text,                                -- sent | delivered | opened | clicked | bounced (события из вебхуков Unisender Go)
  sent_at timestamptz default now()
);
create table events(                          -- продуктовая воронка, in-house аналитика
  id uuid primary key default gen_random_uuid(),
  audit_id uuid references audits(id),
  lead_id uuid references leads(id),
  step text not null,                         -- landed|url_entered|audit_started|teaser_shown|contact_opened|contact_submitted|contact_abandoned|report_viewed|pdf_downloaded|followup_clicked
  meta jsonb,
  created_at timestamptz default now()
);
create index on events(step);
create index on events(created_at);
create index on leads(phone);
create index on leads(email);
create index on leads(created_at);
```

Доступ только через серверный код с service-ключом.

## 4. Обработка аудита (без serverless, фоновый воркер)

1. API `POST /api/audit` валидирует, проверяет лимит, создаёт запись `pending`, возвращает `auditId`.
2. Воркер (отдельный pm2-процесс) берёт `pending` задачи и выполняет пайплайн строго по одной за раз (бережём память под Chromium), после каждого шага пишет `progress`.
3. Фронт опрашивает `GET /api/audit/:id` и при `done` рисует результат. При `error` показывает понятное сообщение.

Пайплайн:
1. Валидация и нормализация URL.
2. Скрапер (раздел 5): десктоп и мобайл скриншоты, отрисованный HTML, текст, структура, наличие schema.
3. Определение типа сайта (ecommerce|leadgen|saas|info|local) и ниши по контенту — лёгкий LLM-вызов. Тип уходит в главный промпт и адаптирует лензу аудита (раздел 6).
4. PageSpeed Insights API (мобайл и десктоп): LCP, CLS, INP, оценка.
5. Проверка ИИ-видимости (AEO, раздел 6.1) по отрисованному HTML: schema, серверный рендер контента, семантика, мета.
6. Поиск конкурентов через встроенный веб-поиск OpenAI (раздел 9), затем тем же скрапером снимаем их.
7. Главный vision-вызов (раздел 7 и 8) с данными целевого сайта, типом сайта, целью и конкурентами.
8. Аннотация скриншотов: на десктоп-скриншот накладываются выноски к ключевым находкам (раздел 13).
9. Генерация PDF отчёта тем же Playwright (print-to-pdf страницы результата), ссылка в `pdf_url`.
10. Сохранение результата, статус `done`. Любой сбой → `error` с понятным текстом, не падать молча. PDF и аннотация — не критичные шаги: если не вышли, отчёт всё равно отдаётся.

## 5. Скрапер (свой, на этом же сервере)

Готовый файл scraper.js (Playwright + Express). Принимает `POST /scrape` с ключом и ссылкой, отдаёт десктоп и мобайл скриншоты (jpeg, сжатые), отрисованный HTML, извлечённые заголовки, кнопки, формы, текст, наличие schema и телефона. Внутри защита от SSRF и очередь по одной странице за раз.

Приложение и скрапер на одном сервере, поэтому вызов локальный. Если сайт блокирует или не читается, скрапер отдаёт мягкую ошибку, аудит переводится в `error` с понятным сообщением. Опциональный запасной вариант на будущее: платный ScreenshotOne, если попадётся сайт, который твой скрапер не берёт.

Перед отправкой скриншотов в vision уменьшать их по размеру, чтобы не жечь токены.

Этот же Playwright переиспользуется для генерации PDF отчёта (print-to-pdf страницы результата) и для аннотации скриншотов выносками — отдельный платный сервис не нужен.

## 6. Методология аудита (ядро)

Аудит ищет деньги, а не придирки. Каждая находка это либо «здесь уходят» (трение, отвлечение), либо «здесь не понимают или не верят» (ценность, ясность, доверие). Всё переводится в влияние на конверсию и ранжируется по силе эффекта (ICE). Для ключевых находок дополнительно даём грубую оценку потери в процентах конверсии или в деньгах (диапазоном, честно, без фейковой точности) — это сердцевина обещания «где сайт теряет деньги».

Ленза адаптируется под тип сайта (определён в пайплайне): интернет-магазин, лид-ген, SaaS, инфопродукт и локальная услуга судятся по-разному (длина, доказательства, тип CTA, что считать трением). Если заполнена цель сайта со стартовой формы — модель использует её для оценки релевантности и message-match.

Каркас, модель LIFT, 6 сил вокруг ценностного предложения: ценностное предложение, релевантность, ясность, тревога, отвлечение, срочность. Конверсия растёт от релевантности, ясности и срочности и падает от отвлечения и тревоги.

Диагностика по 7 направлениям:
1. Ценность и релевантность: понятно ли уникальное предложение, совпадает ли с запросом, конкретика против воды.
2. Первый экран (5 секунд): что это, для кого, что делать, виден один CTA, не стоковый визуал.
3. Структура и логика воронки: один блок одна задача, верный порядок (релевантность, проблема, решение, выгоды, доказательства, оффер, возражения, финальный CTA), доказательства не закопаны, нет перегруза, один главный CTA.
4. Доверие и снятие возражений: качество соцдоказательства, сигналы надёжности, закрыты ли топ возражения, гарантии.
5. Путь к действию и трение: CTA и баннерная слепота, оффер и якорь, трение, длина и UX формы, отвлечение.
6. Мобайл и юзабилити: адаптив, tap targets, зона пальца и sticky-CTA, читаемость и контраст, попапы, навигация.
7. Техника, скорость, SEO-база: PageSpeed и отказы, schema, мета и заголовки.

### 6.1 ИИ-видимость (AEO)

Восьмое направление: видят ли сайт ИИ-поисковики и ассистенты (ChatGPT, Perplexity, Gemini, Яндекс Нейро). ИИ-агенты парсят HTML, а не «смотрят» страницу, поэтому оцениваем по отрисованному HTML: есть ли структурированные данные (schema.org), отдаётся ли контент в HTML или прячется за клиентским рендером, семантическая разметка, корректные мета и заголовки, есть ли в тексте факты о бизнесе. Выдаём отдельный балл `aeo_score` и находки. Это и часть аудита, и сильный хук для лид-магнита («проверь, видит ли твой сайт ИИ-поиск»).

(Заметка на будущее, для себя: AEO-оптимизацию можно вынести в отдельную платную услугу рядом с редизайном.)

Плюс конкурентный разрыв: что есть у большинства конкурентов, но нет у этого сайта.

## 7. Промпт аудита (мозг)

```
Ты senior CRO и UX аудитор уровня сильного агентства.
Анализируешь сайт СТРОГО по предоставленным данным: скриншоты (десктоп и мобайл),
извлечённый текст и структура, метрики скорости, данные конкурентов.
Задача: где и почему сайт теряет деньги.

Каркас LIFT, оцени 6 сил вокруг ценностного предложения:
value_prop, relevance, clarity, anxiety, distraction, urgency.
Конверсия растёт от релевантности, ясности, срочности и падает от отвлечения и тревоги.

Тебе передаются: тип сайта и (если есть) заявленная цель сайта. Учитывай их:
суди сайт по нормам его типа и оценивай релевантность относительно его цели.

Жёсткие правила:
1. Оценивай только то, что видишь в данных. Не выдумывай отсутствующее.
2. Никаких общих советов. Каждый вывод цитирует конкретный элемент ЭТОГО сайта.
3. Для находки: что не так, почему теряются заявки (механика), критичность (high/medium/low),
   и грубая оценка потери (impact_estimate): диапазон в % конверсии или словами про деньги,
   честно, без фейковой точности. Если оценить нельзя — так и скажи.
4. Тон экспертный, прямой, уважительный, без воды.
5. Конкурентный разрыв: что есть у большинства конкурентов, но нет у этого сайта.

Проверь 8 направлений (ценность и релевантность; первый экран за 5 сек;
структура и логика воронки; доверие и возражения; путь к действию и трение;
мобайл и юзабилити; техника, скорость, SEO-база; ИИ-видимость/AEO — видят ли сайт
ИИ-поисковики, по schema, серверному рендеру контента, семантике и мета).

Дай оценки 0-100: conversion_score, usability_score, aeo_score, overall_score и каждую из 6 сил LIFT.
Ранжируй находки по влиянию на деньги. Верни СТРОГО валидный JSON по схеме.
```

## 9. Конкуренты (веб-поиск OpenAI + свой скрапер)

1. Определяем нишу и гео из контента сайта (маленький LLM-вызов).
2. Встроенный веб-поиск OpenAI (web search tool в Responses API) находит 3–5 реальных конкурентов в этой нише и регионе, с источниками — инструмент возвращает ссылки и `url_citation`. Просим источники, отсекаем каталоги и маркетплейсы. Стоит ~$10 за 1000 вызовов плюс найденный контент как входные токены, ~1-2 цента на аудит. (Опционально: при желании поднять качество и при наличии оплаты этот шаг можно подменить на Perplexity Sonar — интерфейс тот же «запрос → список с источниками».)
3. Тем же скрапером снимаем скриншоты и HTML найденных конкурентов, чтобы сравнение было визуальным и фактическим, а не на словах.
4. Vision сравнивает и выдаёт разрыв: что есть у большинства, но нет у целевого сайта.

Graceful fallback: если нормальных конкурентов не нашли или их не удалось снять, аудит всё равно выдаётся без этого блока, не падает. Конкурентов жёстко ограничить 3–5, чтобы не жечь время и токены.

## 10. Лимит: один бесплатный аудит на человека в месяц

На `POST /api/audit` до запуска проверяем: есть ли в `leads` запись с этим контактом (нормализованный телефон, телеграм или почта) за последние 30 дней. Если есть, мягко отказываем: «Вы уже проверяли сайт в этом месяце. Следующий бесплатный через N дней, или напишите мне для разбора». IP как вторичный предохранитель против обхода. Лимит в конфиге, чтобы легко менять.

Кнопку «проверить ещё сайт» убрать или вести на связь с тобой, чтобы не обещать безлимит.

## 11. Захват лида, подарок, доставка

- Форма: телефон или телеграм (почта по желанию) плюс согласие. Без выбора канала, одно поле контакта и одна кнопка «Получить разбор и подарок».
- Подарок: business-facing файл (чек-лист «Где сайт теряет заявки», уже готов). Лежит в хранилище.
- Отчёт в PDF: кнопка «Скачать PDF» на уже разблокированном полном отчёте. Контакт берём ОДИН раз (тот же гейт, что открывает полный разбор) — повторно за PDF номер НЕ спрашиваем. PDF, подарок и полный разбор — всё это награда за один контакт, чтобы гейт был ценным.
- Доставка по каналам:
  - На экране результата ссылка на подарок видна сразу (работает всегда).
  - Telegram: бот НЕ может написать первым тому, кто его не запускал. Поэтому на экране кнопка «Забрать в Telegram» с диплинком t.me/ИМЯ_БОТА?start=ТОКЕН. Человек жмёт, бот по токену находит лид и отдаёт подарок и ссылку на отчёт.
  - СМС (через провайдера) и почта (через Unisender Go, транспорт) пушатся напрямую, со ссылкой. Шаблоны писем свои (в коде, под бренд), Unisender Go только доставляет.
- Логируем отправки в emails_log, не дублируем. События воронки (открыл форму, отправил, бросил и т.д.) пишем в events.

## 12. Follow-up (ведёт к заказу редизайна)

Реализуется in-house: отдельный крон в воркере (например, раз в час) берёт лидов, которым пора следующее касание (по `followup_stage` и `created_at`), отправляет письмо через Unisender Go, увеличивает `followup_stage`, пишет в emails_log. Шаблоны писем свои, под бренд (см. docs/DESIGN.md). Никакого внешнего конструктора — только транспорт.

- Почта (если есть), 3 касания: день 0 отчёт и подарок, день 2–3 «с чего я бы начала» и оффер, день 6–7 кейс и оффер.
- Телеграм-бот (после запуска человеком): сразу подарок, потом 1–2 догоняющих сообщения с оффером.
- В каждом касании одно действие: написать тебе или заказать редизайн.
- Клик по ссылке в письме (вебхук Unisender Go) → событие в emails_log/events, лид помечается `engaged`.

## 14. Безопасность и данные

- Все ключи в env на сервере, не в клиенте.
- SSRF-защита в скрапере (уже есть): только http/https, блок приватных и локальных адресов.
- Rate limiting: один бесплатный аудит на контакт в месяц (раздел 10) плюс лимит по IP.
- Cloudflare Turnstile против ботов.
- Дневной предохранитель на число аудитов и расходы.
- 152-ФЗ: собираешь персональные данные, поэтому нужна явная галочка согласия с корректной формулировкой (согласие на обработку персональных данных) и страница политики конфиденциальности. Хранить минимум: контакт, факт и дату согласия.
- Воронка и аналитика лидов (in-house): закрытая страница `/admin` (под паролем или баз-аутом) читает Supabase и показывает воронку — сколько аудитов прогнали → сколько контактов поймали → сколько прогрелось (engaged) → сколько в клиентов; конверсию между шагами и drop-off по таблице events (где именно отваливаются и можно понять почему). Лиды видны списком со статусом, баллом аудита, типом сайта, каналом и тем, что отправлено. Аналитика писем (доставки, открытия, клики) приходит вебхуками Unisender Go.
- Важно по порядку: события в events логируем с первого дня (дёшево), а сам дашборд с воронкой и drop-off строим, когда пойдёт трафик — иначе рисуем аналитику по нулю данных. Сам аудит, захват и письма первичны.

## 15. Стоимость

- Модели (всё через прямой ключ OpenAI): vision-аудит на GPT класса GPT-5.4 (рабочая, ~$2.50/$15 за млн), GPT-5.5 (~$5/$30) только если на тесте качество не дотянет; мелочь (тип сайта, ниша, AEO) на GPT-5 mini/nano (копейки). Веб-поиск конкурентов ~1-2¢ за аудит. Итого порядка 12-15¢ за аудит.
- Запасной мозг: если на спайке выдача GPT окажется слишком общей, аудит можно перевести на Claude (Sonnet 4.6 ~$3/$15 или Opus 4.8 ~$5/$25) через kie.ai. По цене Claude и GPT равны (Sonnet≈GPT-5.4, Opus≈GPT-5.5), так что это выбор по качеству, а не по деньгам; решать после реального теста.
- Дорогие шаги: vision и снятие конкурентов. Конкурентов 3–5, скриншоты сжимать и уменьшать перед vision.
- Контакт спрашиваем до запуска. Лимит один в месяц сам режет расходы.
- Лимиты в конфиге (макс конкурентов, макс аудитов в день, дневной бюджет).
- Кэш результата по URL на короткое время. Статичную часть промпта кэшировать (prompt caching), чтобы не платить за неё каждый раз.

## 19. Порядок сборки

1. Сервер: Node, nginx, HTTPS, pm2. Завести Supabase, прогнать SQL из раздела 3 (включая goal, site_type, pdf_url, leads.status/followup_stage и таблицу events).
2. Скрапер: поднять scraper.js на сервере, проверить `/scrape` на паре реальных сайтов (включая Tilda). Убедиться, что отдаёт schema и отрисованный HTML (нужно для AEO).
3. Каркас Next.js: лендинг, форма с согласием, Turnstile и НЕобязательным полем «цель сайта», API `POST /api/audit` и `GET /api/audit/:id`, проверка лимита. С этого шага логируем события воронки в events (landed, url_entered, contact_opened и т.д.).
4. Воркер: фоновый процесс, пайплайн на один сайт (скрапер, определение типа сайта, PageSpeed, AEO-проверка по HTML, главный vision по разделам 6–8 с учётом типа и цели, оценки + LIFT + AEO + impact_estimate), прогресс и вывод. Прогнать на 2–3 реальных сайтах, проверить что находки конкретные и оценка потерь адекватная.
5. Конкуренты: встроенный веб-поиск OpenAI, снятие скрапером, матрица, fallback.
6. Захват лида и доставка: бот с диплинком, СМС и почта через Unisender Go (транспорт), свои шаблоны, логирование в emails_log, подарок. Контакт берётся один раз. События — в events.
7. Премиальная выдача (оценки включая AEO, радар, severity, аннотированный скриншот), экран ожидания, лендинг с AEO-хуком, кнопка скачать PDF, дизайн без слопа.
8. PDF-генерация (Playwright print-to-pdf), OG-карточка (satori + resvg), follow-up цепочка (крон в воркере через Unisender Go, шаблоны свои, по followup_stage).
9. Безопасность и лимиты, согласие 152-ФЗ. Страница `/admin`: воронка, конверсия, drop-off по events, список лидов со статусом (строим, когда пойдёт трафик; события логируем с первого дня). Вебхуки Unisender Go → статусы писем в emails_log.
10. Тесты на реальных сайтах, деплой через pm2.

Готово, когда сервис принимает ссылку, гоняет аудит по методологии LIFT и 8 направлениям (включая AEO) с оценками потерь в деньгах и конкурентами, ловит контакт один раз, отдаёт полный разбор + PDF + подарок, делает OG-карточку, держит лимит один в месяц и не падает на тяжёлых сайтах.
