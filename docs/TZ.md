# ТЗ v4: Razbor — сервис AI-аудита сайтов (финал, self-hosted)

Документ для Claude Code. Заменяет v1–v3. Всё приложение живёт на одном выделенном VPS, без Vercel и без serverless-ограничений. База данных на managed Supabase. Скрапер свой (файл scraper.js).

---

## 0. Суть и цель

Название сервиса: **Razbor** (пишется латиницей, читается как «разбор»). Логотип и заголовки капсом: RAZBOR. Слоган-ориентир: «Видно, где сайт теряет заявки».

Сервис по ссылке делает экспертный аудит сайта и отвечает на вопрос: где и почему сайт теряет заявки. Плюс находит конкурентов и показывает, что есть у них, но нет у этого сайта. Это лид-магнит и витрина: человек получает пользу и за контакт получает подарок, а «как чинить» это платная услуга владельца (редизайн и автоматизация).

Два требования к качеству: выдача экспертная и конкретная (привязана к элементам сайта, мыслит деньгами), и сам сервис выглядит на уровне (не нейро-шаблон).

---

## 1. Инфраструктура (где что крутится)

Один выделенный VPS (Xorek Cloud, отдельный от n8n), Ubuntu 22.04/24.04, минимум 2 vCPU и 4 ГБ RAM (комфортно 8), плюс 2–4 ГБ swap. На нём через pm2 крутятся:
- Next.js-приложение (фронт + API), запуск `next start` за nginx.
- Фоновый воркер аудитов (отдельный процесс pm2).
- Скрапер на Playwright (файл scraper.js, локально на этом же сервере).
- Телеграм-бот (для выдачи подарка).

nginx как реверс-прокси, HTTPS через Let's Encrypt (как ты уже делаешь для neiro-arina.ru), отдельный домен или поддомен под Razbor (проверить свободный: getrazbor, razbor.pro, razbor.site, tryrazbor). Деплой: git pull, `npm run build`, `pm2 restart`.

Внешние сервисы: managed Supabase (БД), OpenAI (vision-анализ), Perplexity (поиск конкурентов), Resend (почта), СМС-провайдер для РФ (опционально), Cloudflare Turnstile (анти-бот).

Vercel и QStash не используются: на постоянном сервере фоновый процесс снимает все ограничения по времени.

## 2. Пользовательский флоу

1. Лендинг: поле URL, понятное обещание, кнопка «Проверить сайт».
2. Перед запуском: телефон или телеграм (почта по желанию), галочка согласия, Turnstile. Обещаем подарок.
3. Проверка лимита: один бесплатный аудит на человека в месяц (раздел 10). Если уже был, мягко сообщаем.
4. Запуск: создаётся задача, фронт показывает экран прогресса с этапами и живым превью скриншота их сайта. Опрашивает статус раз в 2–3 сек (или SSE).
5. Результат: тизер сразу (две оценки и топ-3 точки потери), полный разбор открыт, контакт оставлен.
6. Подарок и отчёт: ссылка на подарок видна на экране сразу, плюс уходит по каналу контакта (раздел 11).
7. В конце один CTA на эксперта.
8. Результат по уникальной ссылке, делится с красивой OG-карточкой (раздел 13).
9. Через несколько дней follow-up (раздел 12).

## 3. База данных (Supabase) + SQL

```sql
create table audits(
  id uuid primary key default gen_random_uuid(),
  url text not null,
  status text not null default 'pending',   -- pending | running | done | error
  progress text,
  result jsonb,
  screenshots jsonb,                          -- ссылки или ключи на скриншоты
  error_message text,
  created_at timestamptz default now()
);
create table leads(
  id uuid primary key default gen_random_uuid(),
  audit_id uuid references audits(id),
  phone text, telegram text, email text,
  channel text,                               -- основной канал контакта
  consent boolean default false, consent_at timestamptz,
  ip text, source text,
  created_at timestamptz default now()
);
create table emails_log(
  id uuid primary key default gen_random_uuid(),
  lead_id uuid references leads(id),
  channel text,                               -- telegram | sms | email
  type text,                                  -- gift | report | followup
  status text,
  sent_at timestamptz default now()
);
create index on leads(phone);
create index on leads(email);
create index on leads(created_at);
```

Доступ только через серверный код с service-ключом.

## 4. Обработка аудита (без serverless, фоновый воркер)

1. API `POST /api/audit` валидирует, проверяет лимит, создаёт запись `pending`, возвращает `auditId`.
2. Воркер (отдельный pm2-процесс) берёт `pending` задачи и выполняет пайплайн строго по одной за раз (бережём память под Chromium), после каждого шага пишет `progress`.
3. Фронт опрашивает `GET /api/audit/:id` и при `done` рисует результат. При `error` показывает понятное сообщение.

Пайплайн:
1. Валидация и нормализация URL.
2. Скрапер (раздел 5): десктоп и мобайл скриншоты, отрисованный HTML, текст, структура.
3. PageSpeed Insights API (мобайл и десктоп): LCP, CLS, INP, оценка.
4. Поиск конкурентов через Perplexity (раздел 9), затем тем же скрапером снимаем их.
5. Главный vision-вызов (раздел 7 и 8) с данными целевого сайта и конкурентов.
6. Сохранение результата, статус `done`. Любой сбой → `error` с понятным текстом, не падать молча.

---

## 5. Скрапер (свой, на этом же сервере)

Готовый файл scraper.js (Playwright + Express). Принимает `POST /scrape` с ключом и ссылкой, отдаёт десктоп и мобайл скриншоты (jpeg, сжатые), отрисованный HTML, извлечённые заголовки, кнопки, формы, текст, наличие schema и телефона. Внутри защита от SSRF и очередь по одной странице за раз.

Приложение и скрапер на одном сервере, поэтому вызов локальный. Если сайт блокирует или не читается, скрапер отдаёт мягкую ошибку, аудит переводится в `error` с понятным сообщением. Опциональный запасной вариант на будущее: платный ScreenshotOne, если попадётся сайт, который твой скрапер не берёт.

Перед отправкой скриншотов в vision уменьшать их по размеру, чтобы не жечь токены.

## 6. Методология аудита (ядро)

Аудит ищет деньги, а не придирки. Каждая находка это либо «здесь уходят» (трение, отвлечение), либо «здесь не понимают или не верят» (ценность, ясность, доверие). Всё переводится в влияние на конверсию и ранжируется по силе эффекта (ICE).

Каркас, модель LIFT, 6 сил вокруг ценностного предложения: ценностное предложение, релевантность, ясность, тревога, отвлечение, срочность. Конверсия растёт от релевантности, ясности и срочности и падает от отвлечения и тревоги.

Диагностика по 7 направлениям:
1. Ценность и релевантность: понятно ли уникальное предложение, совпадает ли с запросом, конкретика против воды.
2. Первый экран (5 секунд): что это, для кого, что делать, виден один CTA, не стоковый визуал.
3. Структура и логика воронки: один блок одна задача, верный порядок (релевантность, проблема, решение, выгоды, доказательства, оффер, возражения, финальный CTA), доказательства не закопаны, нет перегруза, один главный CTA.
4. Доверие и снятие возражений: качество соцдоказательства, сигналы надёжности, закрыты ли топ возражения, гарантии.
5. Путь к действию и трение: CTA и баннерная слепота, оффер и якорь, трение, длина и UX формы, отвлечение.
6. Мобайл и юзабилити: адаптив, tap targets, зона пальца и sticky-CTA, читаемость и контраст, попапы, навигация.
7. Техника, скорость, SEO-база: PageSpeed и отказы, schema, мета и заголовки.

Плюс конкурентный разрыв: что есть у большинства конкурентов, но нет у этого сайта.

## 7. Промпт аудита (мозг)

```
Ты senior CRO и UX аудитор уровня сильного агентства.
Анализируешь сайт СТРОГО по предоставленным данным: скриншоты (десктоп и мобайл),
извлечённый текст и структура, метрики скорости, данные конкурентов.
Задача: где и почему сайт теряет деньги.

Каркас LIFT, оцени 6 сил вокруг ценностного предложения:
value_prop, relevance, clarity, anxiety, distraction, urgency.
Конверсия растёт от релевантности, ясности, срочности и падает от отвлечения и тревоги.

Жёсткие правила:
1. Оценивай только то, что видишь в данных. Не выдумывай отсутствующее.
2. Никаких общих советов. Каждый вывод цитирует конкретный элемент ЭТОГО сайта.
3. Для находки: что не так, почему теряются заявки (механика), критичность (high/medium/low).
4. Тон экспертный, прямой, уважительный, без воды.
5. Конкурентный разрыв: что есть у большинства конкурентов, но нет у этого сайта.

Проверь 7 направлений (ценность и релевантность; первый экран за 5 сек;
структура и логика воронки; доверие и возражения; путь к действию и трение;
мобайл и юзабилити; техника, скорость, SEO-база).

Дай оценки 0-100: conversion_score, usability_score, overall_score и каждую из 6 сил LIFT.
Ранжируй находки по влиянию на деньги. Верни СТРОГО валидный JSON по схеме.
```

```json

  "lift":{"value_prop":0,"relevance":0,"clarity":0,"anxiety":0,"distraction":0,"urgency":0},

```

Полный разбор (после контакта): areas, competitor_gaps, detailed_fixes.

## 9. Конкуренты (Perplexity + свой скрапер)

1. Определяем нишу и гео из контента сайта (маленький LLM-вызов).
2. Perplexity API (Sonar) находит 3–5 реальных конкурентов в этой нише и регионе, с ссылками. Просим источники, отсекаем каталоги и маркетплейсы.
3. Тем же скрапером снимаем скриншоты и HTML найденных конкурентов, чтобы сравнение было визуальным и фактическим, а не на словах.
4. Vision сравнивает и выдаёт разрыв: что есть у большинства, но нет у целевого сайта.

Graceful fallback: если нормальных конкурентов не нашли или их не удалось снять, аудит всё равно выдаётся без этого блока, не падает. Конкурентов жёстко ограничить 3–5, чтобы не жечь время и токены.

---

## 10. Лимит: один бесплатный аудит на человека в месяц

На `POST /api/audit` до запуска проверяем: есть ли в `leads` запись с этим контактом (нормализованный телефон, телеграм или почта) за последние 30 дней. Если есть, мягко отказываем: «Вы уже проверяли сайт в этом месяце. Следующий бесплатный через N дней, или напишите мне для разбора». IP как вторичный предохранитель против обхода. Лимит в конфиге, чтобы легко менять.

Кнопку «проверить ещё сайт» убрать или вести на связь с тобой, чтобы не обещать безлимит.

---

## 11. Захват лида, подарок, доставка

- Форма: телефон или телеграм (почта по желанию) плюс согласие. Без выбора канала, одно поле контакта и одна кнопка «Получить разбор и подарок».
- Подарок: business-facing файл (чек-лист «Где сайт теряет заявки», уже готов). Лежит в хранилище.
- Доставка по каналам:
  - На экране результата ссылка на подарок видна сразу (работает всегда).
  - Telegram: бот НЕ может написать первым тому, кто его не запускал. Поэтому на экране кнопка «Забрать в Telegram» с диплинком t.me/ИМЯ_БОТА?start=ТОКЕН. Человек жмёт, бот по токену находит лид и отдаёт подарок и ссылку на отчёт.
  - СМС (через провайдера) и почта (Resend) пушатся напрямую, со ссылкой.
- Логируем отправки в emails_log, не дублируем.

## 12. Follow-up (ведёт к заказу редизайна)

- Почта (если есть), 3 касания: день 0 отчёт и подарок, день 2–3 «с чего я бы начала» и оффер, день 6–7 кейс и оффер.
- Телеграм-бот (после запуска человеком): сразу подарок, потом 1–2 догоняющих сообщения с оффером.
- В каждом касании одно действие: написать тебе или заказать редизайн.

## 13. Дизайнерские усиления

1. Вирусная OG-карточка результата. Вне Vercel генерируем через satori (HTML в SVG) плюс @resvg/resvg-js (SVG в PNG): имя сайта, оценки Конверсия и Удобство, логотип RAZBOR, фирменный editorial-стиль. Делится с красивой карточкой.
2. Премиальная выдача: шкалы двух оценок, мини-радар по 6 силам LIFT, цветовая система severity, заголовок со скриншотом и именем сайта, матрица конкурентов, план правок с метками effort и эффекта. Палитра оксблад, бумага, чёрный, заголовки Unbounded, текст Manrope.
3. Экран ожидания с предвкушением: этапы и живое превью скриншота.
4. Тон отчёта: экспертный, честный, не убивающий. Сначала что хорошо, потом дыры, потом план.

## 14. Безопасность и данные

- Все ключи в env на сервере, не в клиенте.
- SSRF-защита в скрапере (уже есть): только http/https, блок приватных и локальных адресов.
- Rate limiting: один бесплатный аудит на контакт в месяц (раздел 10) плюс лимит по IP.
- Cloudflare Turnstile против ботов.
- Дневной предохранитель на число аудитов и расходы.
- 152-ФЗ: собираешь персональные данные, поэтому нужна явная галочка согласия с корректной формулировкой (согласие на обработку персональных данных) и страница политики конфиденциальности. Хранить минимум: контакт, факт и дату согласия.
- Простая воронка в БД: сколько аудитов, сколько контактов, сколько ответов на follow-up, чтобы мерить, что работает.

## 15. Стоимость

- Дорогие шаги: vision и снятие конкурентов. Конкурентов 3–5, скриншоты сжимать и уменьшать перед vision.
- Контакт спрашиваем до запуска. Лимит один в месяц сам режет расходы.
- Лимиты в конфиге (макс конкурентов, макс аудитов в день, дневной бюджет).
- Кэш результата по URL на короткое время.

## 19. Порядок сборки

1. Сервер: Node, nginx, HTTPS, pm2. Завести Supabase, прогнать SQL из раздела 3.
2. Скрапер: поднять scraper.js на сервере, проверить `/scrape` на паре реальных сайтов (включая Tilda).
3. Каркас Next.js: лендинг, форма с согласием и Turnstile, API `POST /api/audit` и `GET /api/audit/:id`, проверка лимита.
4. Воркер: фоновый процесс, пайплайн на один сайт (скрапер, PageSpeed, главный vision по разделам 6–8, две оценки и LIFT), прогресс и вывод.
5. Конкуренты: Perplexity-поиск, снятие скрапером, матрица, fallback.
6. Захват лида и доставка: бот с диплинком, СМС и Resend, логирование, подарок.
7. Премиальная выдача, экран ожидания, лендинг, дизайн без слопа.
8. OG-карточка (satori + resvg) и follow-up цепочки.
9. Безопасность и лимиты, согласие 152-ФЗ, воронка в БД.
10. Тесты на реальных сайтах, деплой через pm2.

Готово, когда сервис принимает ссылку, гоняет аудит по методологии LIFT и 7 направлениям с двумя оценками и конкурентами, ловит контакт, отдаёт подарок по нужному каналу, делает OG-карточку, держит лимит один в месяц и не падает на тяжёлых сайтах.
