# ТЗ: раздел блога /blog для Razbor

Блог под SEO/AEO-статьи (контент-план). Цель — чистое удобное чтение, сильная
типографика, быстрый, и чтобы статьи хорошо индексировались и цитировались
нейросетями. В стиле Razbor (оксблад/бумага, Unbounded/Manrope, brutalism, без
слопа). Сверяться с docs/BUILD-GUIDE.md и docs/ai-seo-guide.md.

Собрано по принципам сильных блогов 2026 (Notion, SavvyCal, Riverside — минимализм,
типографика, контент в центре, одна колонка).

---

## Структура

1. **/blog** — список статей (индекс).
2. **/blog/[slug]** — страница статьи.
3. Статьи хранить как файлы (markdown в репо) или в базе — на выбор разработчика,
   проще начать с markdown-файлов в проекте (контент редко меняется, легко
   версионировать в git). Каждая статья = .md с frontmatter (title, description,
   slug, date, заголовки для SEO).

---

## Страница статьи /blog/[slug] — ГЛАВНОЕ (тут читают)

**Раскладка (по консенсусу сильных блогов 2026):**
- ОДНА колонка, контент по центру. Без сайдбаров в статье (тренд 2026, чище на
  мобайле). Это решает проблему «узкой кишки» — текст не растянут и не сжат.
- Ширина текста (reading width) 60-75 символов в строке (примерно max-w-[680px]).
  НЕ шире (тяжело читать), не уже (тесно). Критично для комфорта.
- Body-текст 17-18px, line-height 1.7. Заголовки крупные (Unbounded), но не
  агрессивные.
- Высокий контраст текста (ink на бумаге), как в бренде.
- Абзацы короткие (2-4 предложения), H2/H3 каждые 200-300 слов — разбивка,
  не стена текста.

**Элементы страницы (сверху вниз):**
- Заголовок статьи (H1, крупно).
- Мета-строка: дата + «N минут чтения» (посчитать по объёму). Видно сколько вложить.
- (опционально) Оглавление — для длинных статей, прыгать по разделам. Можно
  sticky сбоку на десктопе ИЛИ свёрнутый блок вверху на мобайле. Снижает отказы.
- Тело статьи: чистая типографика, списки/выделения для скана (только 27% читают
  целиком, 73% сканируют — структура важна).
- CTA по ходу (ненавязчиво, 1 раз в середине) + в конце — «Проверить свой сайт
  бесплатно → getrazbor.ru».
- Блок «Похожие статьи» внизу (2-3 ссылки на соседние) — держат на сайте.
- FAQ-блок если есть в статье (с FAQPage schema).

**SEO/AEO на странице (важно — это цель блога):**
- Чёткая H1 → H2 → H3 иерархия, ответы вверху разделов (для Google AI Overviews
  и ChatGPT — они тянут из чёткой структуры). По ai-seo-guide.
- Article schema (JSON-LD) на каждой статье: заголовок, дата, автор, описание.
- FAQPage schema если есть FAQ.
- Мета title/description из frontmatter, canonical, OG.
- Каждая статья в sitemap.xml (добавлять по мере выхода).
- Внутренние ссылки между статьями (перелинковка) + на главную (запустить разбор).

## Страница списка /blog

- Заголовок раздела + короткое описание («Разбираем, почему сайты теряют заявки
  и как это чинить» — язык боли).
- Список/сетка карточек статей: заголовок, короткий excerpt (15-45 слов), дата,
  «N мин». Uniform-карточки, чисто, без хлама.
- Сортировка новые сверху.
- Чистый минимализм по бренду, не пёстро.
- Ссылка на блог добавить в навигацию/футер сайта.

---

## Технически

- Next.js (как весь сайт). Статьи — markdown в репо (напр. /content/blog/*.md)
  с frontmatter, рендерить через md-парсер. Или в Supabase — но md проще для старта.
- Статика/SSG где можно (блог редко меняется) — быстро, хорошо для SEO.
- Картинки lazy, оптимизированы (если будут).
- Мобайл-first: одна колонка, крупный текст, ничего не уезжает.
- Скорость: лёгкие страницы, без тяжёлых скриптов (Core Web Vitals — фактор
  ранжирования 2026).

## Что НЕ делать
- Не лепить сайдбары в статью (отвлекает, плохо на мобайле).
- Не растягивать текст во всю ширину (нечитаемо).
- Не AI-слоп (Inter, фиолетовый градиент, generic-карточки) — по BUILD-GUIDE.
- Не тяжёлые скрипты/анимации, убивающие скорость.

## Порядок
1. Сначала страница статьи /blog/[slug] (рендер md + типографика + schema) —
   на ней главное, заодно зальём 2 готовые статьи, проверим как читается.
2. Потом /blog (список).
3. Ссылка в навигацию + статьи в sitemap.
4. Скрин десктоп + мобайл, проверить ширину текста и читаемость.
