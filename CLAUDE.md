# AI-аудит сайтов — память проекта

Полное ТЗ лежит в `docs/TZ.md` — это источник правды.
Если что-то здесь противоречит ТЗ, прав ТЗ. Если ТЗ молчит или противоречит сам себе — спроси меня, не придумывай.

## Что строим
Сервис: по ссылке делает экспертный CRO/UX-аудит сайта (где и почему сайт теряет деньги),
находит конкурентный разрыв, отдаёт премиальную выдачу. Это лид-магнит: подарок за контакт,
follow-up ведёт к платной услуге (редизайн). Бесплатная выдача = витрина экспертности владельца.

## Стек (не менять без спроса)
- Next.js App Router + Vercel
- Supabase Postgres — таблицы `audits`, `leads`, `emails_log`
- Upstash QStash — очередь и ретраи (паттерн «задача + опрос»)
- ScreenshotOne — скриншоты desktop + mobile, рендер HTML
- Google PageSpeed Insights API — скорость
- Serper.dev — поиск конкурентов
- OpenAI vision — мозг аудита (имя модели уточнить при сборке, не хардкодить вслепую)
- Telegram-бот + СМС + Resend — доставка подарка/отчёта/follow-up
- @vercel/og — шеринг-карточка
- Cloudflare Turnstile — антибот

## Жёсткие правила (нарушение = стоп и переделка)
1. Все ключи только на сервере, env Vercel. Никогда в клиенте. service-ключ Supabase — только в серверных роутах. Публичный ключ на запись не давать.
2. SSRF-защита на приёме URL обязательна: только http/https, резолвить домен, блокировать localhost, 127.0.0.1, 10.x, 192.168.x, 172.16–31.x, облачные метаданные. Это не «потом», это сразу.
3. Rate limiting по IP и по контакту. Turnstile перед запуском аудита. Дневной лимит на число аудитов и бюджет — в конфиг.
4. Долгий аудит НЕ выполнять внутри одного запроса. Только async: POST создаёт запись → QStash → воркер по шагам пишет progress → фронт опрашивает GET. Иначе таймаут на Vercel.
5. Не логировать секреты и персональные данные.

## Дисциплина по деньгам
Самые дорогие шаги — vision-вызов и анализ конкурентов. Конкурентов жёстко ограничить 3–5.
Кэшировать результат по URL на короткое время. Все лимиты (макс конкурентов, макс аудитов в день,
дневной бюджет, имя модели) — в один конфиг-файл, не разбросаны по коду.

## Конвенции кода
- Работа с БД только через серверные роуты с service-ключом.
- Env через `process.env` на сервере, всё задокументировать в `.env.example`.
- Конфиг лимитов и моделей — отдельный файл, не хардкод.
- Промпт аудита и JSON-схема выдачи — в отдельном модуле, чтобы менять без правок логики.

## Качество — на этом всё держится
- Выдача аудита конкретная, привязана к элементам ИМЕННО этого сайта, мыслит деньгами и воронкой.
  Никакого общего AI-текста. Находка вида «добавьте призыв к действию» без цитаты элемента = брак, переделать.
- Каждая находка: что не так (конкретно) → почему теряются заявки (механика) → сила влияния (high/medium/low).
- Интерфейс премиальный, без шаблонного нейро-дизайна. Применять фильтр против AI-слопа.

## Рабочий процесс (соблюдать)
- Перед коммитом: `npm run build` должен проходить. Красное не коммитим.
- После `git push`: проверить, что деплой на Vercel в статусе READY, прежде чем считать задачу сделанной.
- Один логический кусок = один коммит. Коммитим часто.
- Не делать всё ТЗ за один проход. Идём по фазам из `BUILD-PLAN.md`, строго по одной, с остановкой на проверку.
- Сначала план (plan mode), я подтверждаю, потом код.

## Навыки под задачи

## Definition of done
Сервис принимает ссылку, выдаёт экспертный разбор по LIFT и 7 направлениям с двумя оценками,
анализом структуры воронки и матрицей конкурентов, дарит подарок за контакт по нужному каналу,
делает красивую OG-карточку и не падает на длинных аудитах.
