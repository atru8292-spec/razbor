# Razbor — память проекта

Источник истины: `docs/TZ.md` (ТЗ v7). Стартовый бриф и приёмка: `docs/BRIEF.md`, дизайн: `docs/DESIGN.md`.
Перед работой прочитать. Если что-то здесь противоречит ТЗ — прав ТЗ, спроси меня. Текст ТЗ тут не дублируем: только durable-правила, детали — в `docs/TZ.md`.

## Что это
Razbor — сервис AI-аудита сайтов: по ссылке показывает, где сайт теряет заявки.
Лид-магнит; за один контакт — полный разбор, PDF-отчёт и подарок (чек-лист); follow-up ведёт к платному редизайну.
Имя латиницей, в логотипе и заголовках капсом: RAZBOR. Слоган: «Видно, где сайт теряет заявки».

## Архитектура — НЕ Vercel, НЕ serverless
Всё на одном VPS через pm2: Next.js (`next start` за nginx), фоновый воркер аудитов (отдельный процесс),
скрапер на Playwright (scraper.js, локально на том же сервере), Telegram-бот. nginx + Let's Encrypt.
Managed Supabase. Деплой: git pull → npm run build → pm2 restart.

Внешние сервисы (v7): весь ИИ — OpenAI напрямую (vision-аудит на GPT класса 5.4, мелочь на mini/nano,
конкуренты — встроенный веб-поиск OpenAI в Responses API). Почта — Unisender Go (только транспорт API/SMTP).
Google PageSpeed, Cloudflare Turnstile, СМС-провайдер (опционально). Perplexity и Claude (через kie.ai) —
опциональные апгрейды по качеству после теста, не обязательны.

ЗАПРЕЩЕНО: Vercel, QStash, платный скриншот-API (скрапер свой), `@vercel/og`
(OG-карту делаем через satori + resvg), OpenRouter (в РФ заблочен). Конструктор писем/сегментацию/
аналитику у Unisender НЕ берём — пишем сами.

## Жёсткие правила (нарушение = стоп и переделка)
1. Все ключи только в env на сервере, никогда в клиенте. Supabase service-ключ — только в серверном коде.
2. SSRF в скрапере: только http/https, блок приватных и локальных адресов.
3. Лимит: один бесплатный аудит на контакт в месяц + лимит по IP. Turnstile перед запуском. Дневной предохранитель на число аудитов и бюджет.
4. 152-ФЗ: явная галочка согласия на обработку ПДн + страница политики. Хранить минимум: контакт, факт и дата согласия.
5. Аудит — в фоновом воркере, по одной задаче за раз (бережём память под Chromium). Сервер постоянный, обходы таймаутов не нужны.
6. Везде мягкие ошибки. Не падать молча: сбой → audit.status = error с понятным текстом. PDF и аннотация — не критичные шаги: не вышли → отчёт всё равно отдаётся.
7. Контакт берётся ОДИН раз: полный разбор, PDF и подарок — награда за этот один контакт, повторно номер не спрашиваем.
8. События воронки пишем в таблицу `events` с первого дня (дёшево). Закрытая `/admin` (баз-аут) с воронкой/drop-off строится, когда пойдёт трафик.

## Качество — на этом всё держится
- Выдача конкретная, привязана к элементам ИМЕННО этого сайта, мыслит деньгами и воронкой.
  Промпт и методологию (LIFT + 8 направлений, включая ИИ-видимость/AEO) из разделов 6–8 ТЗ брать дословно.
  Ключевые находки дают грубую оценку потери (в % конверсии или деньгах, честно, без фейковой точности).
  Лензу адаптировать под тип сайта (ecommerce|leadgen|saas|info|local) и под необязательную «цель сайта» с формы.
  Находка вида «добавьте призыв к действию» без цитаты элемента = брак, переделать.
- Дизайн без AI-слопа: палитра оксблад / бумага / чёрный, заголовки Unbounded, текст Manrope, editorial.
  Никаких фиолетовых градиентов, трёх одинаковых карточек, дефолтного Inter. Применять frontend-design.

## Частые ошибки — не повторять
Полный список (12 пунктов с причинами) — `docs/HANDOFF.md`, раздел 4 «ГРАБЛИ». Кратко:
- ВСЕГДА `git checkout package-lock.json` перед `git pull` (иначе деплой падает на local changes).
- Коммитить по этапам, не один гигантский коммит (легче откатывать и отлаживать).
- ВСЕГДА push после commit — иначе деплой тянет старый код. Проверять `git log --oneline -1` на сервере.
- PageSpeed без ключа отдаёт 429 → `PAGESPEED_API_KEY` в .env воркера; без ключа блок скорости скрываем, не выдумываем.
- Скрапер слушает :8080, не :8090. `SCRAPER_URL=http://localhost:8080`.
- Деплой — внутри `screen -S deploy` (SSH рвётся на долгом build), при обрыве `screen -r deploy`.
